# PRD
**OpenClaw Visual Installer（Windows / WSL-first）MVP 产品需求文档**

No critical clarification needed.

## 1. 背景 / 问题

OpenClaw 在 Windows 环境下的安装门槛较高。普通用户即使接受桌面软件安装，也往往无法稳定完成 WSL 环境准备、运行时初始化、首次启动和失败排查。

当前问题集中在以下方面：
- 用户必须理解过多技术概念，才能完成安装。
- 用户无法快速判断失败发生在系统环境、WSL、发行版还是 OpenClaw 本身。
- 手动安装路径难以标准化，导致支持成本高、环境一致性差、成功率不稳定。

本项目需要提供一个独立的 Windows 图形化安装器，将 OpenClaw 的 WSL 安装路径产品化、标准化、可恢复化。

## 2. 目标

### 2.1 产品目标
- 系统 MUST 让 Windows 用户通过 GUI 完成 OpenClaw 的 WSL 安装。
- 系统 MUST 在 happy path 中尽量避免要求用户手动进入终端。
- 系统 MUST 明确展示安装状态、失败位置和下一步建议。
- 系统 MUST 在安装完成后提供基础运行控制能力。
- 系统 MUST 降低 Windows 场景下的安装支持成本。

### 2.2 成功标准
- 系统 MUST 在受支持环境中实现高安装完成率。
- 系统 MUST 让用户在安装完成后可直接进入 OpenClaw 本地入口。
- 系统 MUST 在失败场景中输出可执行的恢复路径，而不是模糊报错。
- 系统 MUST 使 Windows 安装路径相比手动流程更一致、更可支持。

### 2.3 非目标
- 系统 MUST NOT 在 MVP 中承担完整运行期管理 GUI 的职责。
- 系统 MUST NOT 在 MVP 中支持 Windows Native 运行模式。
- 系统 MUST NOT 在 MVP 中承担多节点管理、插件管理或高级配置编辑职责。
- 系统 MUST NOT 在 MVP 中优先支持高度自定义安装流程。

## 3. 系统类型分类

**分类：infra / installer system（基础设施 / 安装器系统）**

该分类意味着本 PRD 的重点必须放在以下方面：
- 环境兼容性与前置条件识别
- 安装与初始化约束
- 失败恢复能力
- 已有状态的安全处理
- 运行可验证性
- 日志与诊断可支持性

本 PRD MUST 以安装成功率、可恢复性和可支持性为核心，而不是内部实现细节。

## 4. 用户场景 / 用户流程

### 场景 A：用户未安装 WSL
- 系统 MUST 识别 WSL 缺失或未正确启用。
- 系统 MUST 为用户提供完成前置准备所需的明确引导。
- 系统 MUST 在前置条件满足后完成推荐发行版准备、OpenClaw 安装与首次启动验证。
- 系统 MUST 在验证通过后向用户提供成功状态与本地入口。

### 场景 B：用户已安装 WSL，但未安装 OpenClaw
- 系统 MUST 识别可用 WSL 环境。
- 系统 MUST 校验推荐发行版与安装前置条件。
- 系统 MUST 在条件满足后完成安装与首次启动验证。
- 系统 MUST 在验证通过后向用户提供本地入口。

### 场景 C：用户机器上已有 OpenClaw
- 系统 MUST 识别已有安装痕迹。
- 系统 MUST 在处理已有状态前向用户说明当前情况与可选路径。
- 系统 MUST 对修复、复用、升级或重装等路径保持清晰边界。
- 系统 MUST 在执行后向用户输出结果与当前状态。

### 场景 D：安装失败
- 系统 MUST 在检查、准备、安装或启动失败时保留失败上下文。
- 系统 MUST 展示失败步骤、可能原因和下一步建议。
- 系统 MUST 提供重试、修复或导出诊断的入口。

### 场景 E：安装后的轻量控制
- 系统 MUST 在再次启动时识别当前安装与运行状态。
- 系统 MUST 提供启动、停止、重启、打开入口、查看日志和修复等基础操作。
## 5. 功能需求

### 5.1 环境识别与前置条件
- 系统 MUST 检测 Windows 版本是否在支持范围内。
- 系统 MUST 检测 WSL 是否可用。
- 系统 MUST 检测 WSL2 是否满足要求。
- 系统 MUST 检测推荐 Linux 发行版是否存在且可用。
- 系统 MUST 检测目标环境中是否已存在 OpenClaw 安装痕迹。
- 系统 MUST 检测 OpenClaw 当前是否处于运行状态。
- 系统 MUST 检测关键端口或关键资源是否存在明显冲突。
- 系统 MUST 以结构化结果输出上述检测结果。
- 系统 MUST NOT 仅以原始命令输出作为用户可见状态。

### 5.2 安装范围与目标环境
- 系统 MUST 在 MVP 中仅支持 WSL-first 安装模式。
- 系统 MUST 在 MVP 中仅支持一个明确推荐的 Linux 发行版作为默认路径。
- 系统 MUST 对默认安装路径保持一致性。
- 系统 MUST NOT 要求普通用户在主流程中理解多种运行模式差异。
- 系统 SHOULD 将高级安装选项隐藏在非默认路径中。

### 5.3 图形化安装引导
- 系统 MUST 提供完整的图形化安装引导流程。
- 系统 MUST 包含欢迎、检查、准备、确认、执行、成功、失败等核心页面或等价阶段。
- 系统 MUST 让用户始终知道当前处于哪个阶段。
- 系统 MUST 将真实执行进度映射到界面状态。
- 系统 MUST 在需要管理员权限、系统确认或重启时提前提示用户。

### 5.4 WSL 准备与引导能力
- 系统 MUST 识别 WSL 缺失、未启用或状态异常。
- 系统 MUST 为阻塞性前置条件提供明确的人类可读说明。
- 系统 MUST 在可自动完成的情况下完成前置准备。
- 系统 MUST 在无法自动完成的情况下提供最短路径的人工指引。
- 系统 SHOULD 支持在中断或重启后恢复到可继续状态。

### 5.5 OpenClaw 安装与初始化
- 系统 MUST 在推荐 WSL 环境中完成 OpenClaw 安装。
- 系统 MUST 完成 OpenClaw 的基础初始化，使其进入可启动状态。
- 系统 MUST 验证安装结果，而不是仅依赖脚本退出码。
- 系统 MUST 将部分完成或不完整状态识别为失败、未完成或可修复状态。
- 系统 MUST NOT 将未经验证的安装结果标记为成功。

### 5.6 首次启动与可访问性验证
- 系统 MUST 在安装完成后支持首次启动。
- 系统 MUST 验证 OpenClaw 是否真正进入可用状态。
- 系统 MUST 向用户展示本地入口地址或等价入口信息。
- 系统 MUST 提供直接打开本地入口的能力。
- 系统 MUST NOT 在仅检测到进程存在时就判定安装成功。

### 5.7 安装后的基础控制能力
- 系统 MUST 在已安装状态下提供启动能力。
- 系统 MUST 在已安装状态下提供停止能力。
- 系统 MUST 在已安装状态下提供重启能力。
- 系统 MUST 在已安装状态下提供查看当前运行状态的能力。
- 系统 MUST 在已安装状态下提供打开本地入口的能力。
- 系统 MUST 在已安装状态下提供查看日志的能力。
- 系统 MUST 在已安装状态下提供修复入口。

### 5.8 重试、修复与恢复
- 系统 MUST 提供失败后的重试能力。
- 系统 MUST 提供基础修复能力，用于恢复不完整或异常状态。
- 系统 MUST 在修复后明确告知用户修复结果和下一步建议。
- 系统 MUST NOT 要求用户先手动清理大量残留才能重新尝试。
- 系统 SHOULD 在重复运行安装器时保持行为可预测。

### 5.9 已有安装处理
- 系统 MUST 识别已有 OpenClaw 安装痕迹。
- 系统 MUST 在处理已有安装前明确提示用户当前状态。
- 系统 MUST 在覆盖、重装、重置或升级前要求用户确认。
- 系统 MUST NOT 在未提示的情况下覆盖已有安装。
- 系统 SHOULD 在可行时提供修复或复用路径。

### 5.10 日志与诊断
- 系统 MUST 记录环境检查、准备、安装、启动验证和修复过程中的关键日志。
- 系统 MUST 允许用户查看最近日志。
- 系统 MUST 允许用户复制日志或导出诊断信息。
- 系统 MUST 在失败页面中提供与当前失败相关的日志入口。
- 系统 MUST 使诊断导出至少包含环境摘要、关键步骤日志和失败摘要。

### 5.11 错误提示
- 系统 MUST 对常见失败类型提供分类明确的错误提示。
- 系统 MUST 覆盖至少以下错误类型：环境不支持、权限不足、WSL 异常、发行版异常、安装失败、启动失败、端口冲突、资源获取失败。
- 系统 MUST 在每个错误场景中说明失败步骤。
- 系统 MUST 在每个错误场景中说明可能原因。
- 系统 MUST 在每个错误场景中说明推荐下一步。
- 系统 MUST NOT 将“未知错误”作为默认主要错误体验。

### 5.12 安全与破坏性操作
- 系统 MUST 在重装、重置、覆盖已有状态或执行潜在破坏性修复前要求确认。
- 系统 MUST 让用户明确区分重试、修复、重装和重置。
- 系统 MUST NOT 默认执行潜在破坏性操作。

## 6. 非功能需求

### 6.1 可靠性
- 系统 MUST 支持重复运行后的可预测行为。
- 系统 MUST 尽量保证关键安装路径具备幂等性。
- 系统 SHOULD 支持安装中断后的再次进入与恢复。

### 6.2 易用性
- 系统 MUST 使用面向普通用户的文案。
- 系统 MUST NOT 要求普通用户理解 WSL 底层概念才能完成 happy path。
- 系统 MUST 减少主路径中的术语密度与决策点。

### 6.3 性能与响应
- 系统 SHOULD 在合理时间内完成前置检查。
- 系统 MUST 在长时间步骤中展示持续反馈。
- 系统 SHOULD 在合理时间内完成启动验证。

### 6.4 可支持性
- 系统 MUST 生成带时间信息和步骤信息的诊断材料。
- 系统 MUST 让支持方能够基于导出内容快速定位失败阶段。
- 系统 SHOULD 让常见错误具备稳定分类，以支持后续支持体系和统计分析。

### 6.5 安全性
- 系统 MUST 明确提示需要管理员权限的操作。
- 系统 MUST NOT 在日志中默认暴露敏感配置或凭证。
- 系统 MUST NOT 在正常流程中执行不受控的用户自定义脚本。

### 6.6 可维护性
- 系统 MUST 保持产品主流程与内部实现方式解耦。
- 系统 MUST 保持对未来独立管理 GUI 项目的可复用空间。
- 系统 MUST NOT 在 PRD 层预设内部模块交互、执行管线或分层实现。

## 7. 边界情况

- 系统 MUST 处理“WSL 已安装但状态异常”的情况，并将其识别为环境异常而不是普通安装失败。
- 系统 MUST 处理“用户权限不足”的情况，并给出明确下一步。
- 系统 MUST 处理“存在非推荐发行版”的情况，并明确其是否属于支持路径。
- 系统 MUST 处理“机器上已有手动安装的 OpenClaw”的情况，并对接管、修复或重装给出清晰边界。
- 系统 MUST 处理“安装过程中断”的情况，并在再次启动时识别残留状态。
- 系统 MUST 处理“端口冲突”的情况，并说明冲突影响与建议动作。
- 系统 MUST 处理“资源下载或获取失败”的情况，并支持合理重试路径。
- 系统 MUST 处理“进程存在但服务不可用”的情况，并以可访问性或等价可用性验证为准。
- 系统 MUST 处理“用户重复运行安装器”的情况，并避免状态持续恶化。
- 系统 SHOULD 处理“系统更新导致环境漂移”的情况，并保留修复入口。

## 8. 开放问题 / 风险

### 8.1 开放问题
- MVP MUST 明确支持哪些 Windows 版本和最低构建号。
- MVP MUST 明确哪些 WSL 前置条件可由产品自动处理，哪些必须由用户确认或手动完成。
- MVP MUST 明确推荐发行版的选择。
- MVP MUST 明确 OpenClaw 的安装来源与版本控制边界。
- MVP MUST 明确首版是否包含升级能力，还是仅包含安装与修复。
- 产品 SHOULD 明确是否需要安装成功率相关的匿名统计能力。

### 8.2 主要风险
- WSL 环境差异可能导致安装行为不稳定。
- 权限与重启链路可能破坏流畅体验。
- 接管已有手动环境可能显著抬高边缘案例复杂度。
- 启动验证不足可能导致误判成功。
- 范围蔓延可能使 installer 偏离其核心职责。

## 9. 后续迭代

- 后续版本 SHOULD 提供更完整的修复能力。
- 后续版本 SHOULD 提供更稳定的升级路径。
- 后续版本 SHOULD 提供更强的启动健康检查能力。
- 后续版本 SHOULD 提供更完善的错误分类与支持导出能力。
- 后续版本 SHOULD 为未来独立管理 GUI 复用核心能力预留接口边界。
- 后续版本 SHOULD 在需求明确后再评估高级安装模式、多发行版支持和 Windows Native 路线。

## 10. 假设

以下内容为显式假设：

- MVP MUST 只支持一个推荐 Linux 发行版。
- MVP MUST 只支持 WSL-first 路线。
- 完整管理 GUI MUST 作为独立项目推进，而不是并入本项目。
- 本项目 MUST 优先优化安装成功率、可恢复性和可支持性，而不是灵活性。

## Architect Handoff Contract

### 1. System Goals (non-negotiable)
- 安装器 MUST 让受支持的 Windows 用户通过 GUI 完成 OpenClaw 的 WSL 安装与首次启动。
- 安装器 MUST 在主路径中尽量避免要求用户手动使用终端。
- 安装器 MUST 对失败场景提供可执行的恢复路径。
- 安装器 MUST 在已安装后提供基础控制与诊断入口。

### 2. Hard Constraints (MUST / MUST NOT)
- 方案 MUST 以 WSL-first 为唯一 MVP 运行路线。
- 方案 MUST 支持一个明确推荐的发行版作为默认路径。
- 方案 MUST NOT 在 MVP 中演化为完整管理 GUI。
- 方案 MUST NOT 依赖用户理解复杂内部概念才能完成主流程。
- 方案 MUST 对已有安装、潜在破坏性操作和失败状态提供明确保护与反馈。
- 方案 MUST 通过可用性验证判断成功，而不是仅通过进程存在或命令退出码判断成功。

### 3. Open Decisions (left for Architect)
- 如何在不改变产品约束的前提下实现环境检查与兼容性判断。
- 如何实现诊断采集、失败分类、恢复机制与可用性验证。
- 如何选择底层技术栈、封装方式和执行通道。
- 如何在不破坏产品边界的前提下为未来管理 GUI 复用能力。

### 4. Flexibility Boundaries
- Architect MAY 自主决定内部结构与实现方式，只要不改变产品目标和约束。
- Architect MAY 自主决定日志、诊断、恢复与验证机制的内部实现方式，只要结果可支持、可验证、可导出。
- Architect MUST NOT 将产品范围扩展到完整管理 GUI、Windows Native 运行或高度自定义安装。

### 5. Success Criteria (machine-checkable)
- 在受支持环境中，系统 MUST 完成环境检查、安装、启动验证和入口提供。
- 在 WSL 缺失、权限不足、端口冲突、安装失败、启动失败等场景中，系统 MUST 输出分类明确的失败结果与下一步建议。
- 在已有安装存在时，系统 MUST 在执行覆盖类动作前要求确认。
- 在安装成功场景中，系统 MUST 仅在完成可用性验证后展示成功状态。
- 在安装后，系统 MUST 提供启动、停止、重启、查看日志和导出诊断等基础能力。

### 6. Risk Priorities
1. 启动成功误判
2. WSL 环境差异导致的不稳定行为
3. 已有安装接管带来的状态混乱
4. 权限和重启流程造成的中断体验
5. 项目范围蔓延导致 MVP 失焦
